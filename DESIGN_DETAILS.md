# 設計詳細 (DESIGN_DETAILS.md)

## クラウドストレージの選定理由

### なぜ Google Drive なのか？
1.  **プライバシーと所有権**: ユーザー自身のクラウド領域にデータを保存するため、開発者がユーザーの個人情報（活動内容）を管理・保持する必要がない。
2.  **無料枠の充実**: ほとんどのユーザーが既に従量課金なしで利用できる。
3.  **App Folder 機能**: アプリ専用の隠しフォルダ (`appDataFolder`) を提供しており、ユーザーの他のファイルに干渉せず、かつユーザーが誤って削除するリスクを低減できる。

## セキュリティ設計

### 認証フロー (OAuth 2.0 with PKCE)
- クライアントサイドのみで安全に認証を完結させるため、PKCE (Proof Key for Code Exchange) の考え方を踏襲した Google Identity Services (GIS) を利用。
- `CLIENT_SECRET` をソースコードに含める必要がなく、静的サイト（GitHub Pages）でも安全に動作する。

### 最小権限の原則
- スコープを `https://www.googleapis.com/auth/drive.appdata` のみに限定。
- アプリは自分が作成したファイル以外（マイドライブ内の写真や文書など）を読み書きすることは技術的に不可能。

## ユーザー体験 (UX) の追求

### サイレントサインインとセッション維持
- リフレッシュのたびにポップアップが出るのは苦痛であるため、`localStorage` に同期有効フラグを保持。
- **sessionStorage の活用**: 取得したアクセストークンを `sessionStorage` に一時保存することで、タブを開いている間のリフレッシュ（F5等）においては、再認証なしで即座に同期を再開できる「リロード・ブリッジ」を実現。
- **自動同期のトリガー (v8-v9)**: 
    - (v8) リロード直後の自動復旧。
    - (v9) **バックグラウンド・ポーリング**: 3分間隔でチェック。
    - (v9) **ウィンドウフォーカス検知**: タブ復帰時やスリープ解除時に即座にチェック。
- **最適化同期アルゴリズム (v9)**: 毎回全データをダウンロードするのではなく、まず Google Drive API のメタデータから `modifiedTime` を取得し、ローカルの `lastSyncedAt` より新しい場合のみ通信を行うことで、トラフィックと負荷を低減。
- **タスク定義とテンプレートの連携強化 (v10)**: 
    - 新規カテゴリ「委員」を追加。
    - 役職フィルタ有効時でも「対象役職：全員（空）」の定型タスクがテンプレートに表示されるよう、表示ロジックを `roleIds.length === 0` の場合にパスするように見直し。
- セキュリティと利便性のバランスを考慮し、ブラウザを閉じればトークンが破棄される `sessionStorage` を採用。

### データ構造の堅牢性 (マイグレーション)
- `AppState` に `version` を持たせることで、プログラムの進化に伴うデータ破壊を防止。
- 読み込み時に常にマイグレーション関数を通すことで、「古いアプリで作成したデータ」を「新しいアプリ」で開いても適切に変換・補完される。

## 役職ベース管理の設計思想

### なぜ「抽出」ではなく「設定」なのか？
当初は全てのタスクを表示し、ユーザーが手動でフィルタリングする案もありましたが、以下の理由で「設定された役職に基づく自動抽出」を採用しました。
- **認知負荷の低減**: 組合役員は多忙であり、自分に関係のないタスクが混在すると優先順位付けの妨げになる。
- **標準化**: 組織として「委員長はこれをする」という定義を共有することで、属人化を防ぐ。
- **例外への配慮**: 「全表示」オプションを設けることで、急な代行業務や全体の把握が必要な場面にも対応可能にした。

## データマイグレーションの設計 (v1 -> v2)

データ構造を破壊せずに新機能を追加するため、以下のルールを徹底しました。
- **デフォルト値の注入**: 新設された `roles` や `definitions` が空の場合、マイグレーション時に標準的な初期セットを自動生成する。
- **IDの保存**: 削除・編集を可能にするため、各定義には `Date.now()` ベースのユニークIDを付与。

## UI/UX の改善履歴

### モバイル版ログインボタンの配置
スマホ版においてログインボタンが画面外になってしまう問題に対し、以下の対応を行いました。
- **ヘッダーへの統合**: サイドバー（ドロワー）内ではなく、常に視認できるトップヘッダーに「ログイン/同期状態」を配置。
- **レスポンシブ・ナビゲーション**: メインタスクへのアクセスをボトムナビ、設定や特殊機能をヘッダーに分けることで、親指での操作性を向上。

### 同期エラーの堅牢化
`TypeError: Failed to fetch` や `Unexpected end of JSON input` などのネットワーク起因のエラーに対し、以下の対策を講じました。
- **詳細なログ出力**: 同期失敗時に、コンソールへ具体的な原因を出力し、ユーザーが再試行すべきか判断しやすくした。
- **ステート管理の分離**: 同期中、成功、エラーの3状態を明示的に UI で表現し、保存中にページを閉じることによるデータ不整合リスクを視覚的に警告。

## クラウド同期フロー詳細

### 同期のオーケストレーション (storage.ts)
`storage.ts` は localStorage と `googleDrive.ts` の仲介役を担います。
1. **プル (syncWithCloud)**: `googleDrive.getFileContent` で最新の JSON を取得。取得後、即座に `migrations.ts` を通してデータ形式を最新版に変換し、localStorage を上書きします。
2. **プッシュ (uploadToCloud)**: 変更があった際、現在の AppState を一つの JSON ファイル `union_app_data.json` としてまとめ、Google Drive へマルチパートアップロードします。

### データの整合性維持
- **Last Write Wins モデル**: 本アプリはシングルユーザー向けのツールであるため、複雑なマージアルゴリズムは避け、最後に保存したデータがクラウドの正解となる単純なモデルを採用しました。
- **自動再試行**: 認証トークンが切れている場合は、自動的にサイレントサインインを試行し、成功後に中断されたアップロードを継続します。
## メモシステムの正規化 (v5-v6)

### なぜグローバル管理なのか？
当初は各予定（`ScheduleEvent`）や各タスク（`Task`）の中に `memos` 配列を持っていましたが、以下の問題がありました。
- **検索性の低さ**: メモを探すために各予定を開く必要がある。
- **再利用性の欠如**: 一つのメモを複数の予定やタスクに関連付けることができない。
- **拡張性の制限**: 音声データなどの重いデータを複数の場所で管理すると、同期の不整合が起きやすい。

これを v5 で `AppState.memos` へ正規化し、v6 でテンプレート機能を追加することで、情報の集約と入力の高速化を両立しました。

## UIのコンテキスト分離 (カレンダー)

### タブUIの採用理由
- カレンダーの詳細パネルにおいて、「予定の内容確認」と「経費の精算作業」はユーザーの思考モードが異なるため、タブで明示的に分離。
- **予定・タスクタブ**: 「今日何をすべきか」の確認に集中。
- **旅費精算タブ**: 「移動の記録と金額の確定」という事務作業に集中。

### 役職フィルタとステータス管理の設計 (v7-v8)
- **動的フィルタ**: アプリ全体の「自分の役職設定」を汚さずに、カレンダー上で一時的に他役職の項目を確認したいニーズに応え、画面固有の `filterRoleId` ステートを導入。
- **ステータス・フィードバック**: 「予定」は単なる時間の確保だけでなく「実施したか」の証跡でもあるため、タスクと同様の `status` 管理を導入。
- **コンテキスト内編集 (v8)**: カレンダー画面から離れずに「予定」だけでなく「タスク」も編集・完了できるよう、サイドパネルに共通の編集 UI を統合。
- **視覚的アフォーダンス**: 完了した項目をグレーアウト・打消し線にすることで、カレンダーを見るだけで「終わっていない予定」に意識を向けられるよう設計。
